GGUF Architecture & Inference Knowledge DocumentTarget Deployment: pocketgemma4 Engine (Rust/Candle Backend)Based on the documentation mapping from the iuliaturc/gguf-docs repository, here is the technical breakdown of the GGUF ecosystem and how it directly impacts the tensor processing pipeline on the Ryzen hardware.1. Decoding the Payload: google_gemma-4-E2B-it-Q4_K_M.ggufThe naming convention of the GGUF file acts as the architectural blueprint for the memory allocations.Q4 (The Base Bit Width): The primary target for the weights is 4 bits per weight (bpw).K (The K-Quant Architecture): The "K" stands for Kawrakov (the developer who implemented it), not K-Means clustering. It indicates a "Double Quantization" scheme.M (The Size Modifier): Stands for Medium. It indicates a strict mixed-precision distribution. In an M quant, critical tensors (like attn_v and attn_q) are kept at a higher quantization grade (e.g., Q5 or Q6) to preserve attention geometry, while the bulk feed-forward network weights drop to Q4. Token embeddings and LayerNorm weights (like the layer_output_scale we just mapped) remain in full F32 or F16 precision.2. The Internal Mechanics: K-Quants vs LegacyWhen loading the model through Candle's dequantization methods, the engine is processing heavily optimized byte-structures called "Super-blocks."Legacy Quants (e.g., Q4_0): Split tensors into blocks of 32 weights, attaching an FP16 scale and offset constant to each block. This creates massive storage overhead.K-Quants: Group eight regular blocks into a single "Super-block" (256 weights total). The individual block scales are quantized down to INT8, and the entire Super-block gets a single pair of FP16 constants.Execution Impact: This structure is why ensuring the tensor shapes are contiguous in memory is absolutely critical. Super-blocks are engineered for sequential memory reads to maximize cache line utilization. If a tensor is sliced (like we saw when testing the .narrow() SWA logic) without calling .contiguous()? afterward, it shatters the Super-block alignment in RAM, causing the execution to spin-lock and bottleneck the CPU.3. The Importance Matrix (Imatrix)The documentation extensively details the "Importance Matrix," an optimization technique that calculates the importance of individual weights using a calibration dataset.The Math: If changing a specific weight dramatically alters the model's output, it is assigned a high importance score based on its activation magnitudes ($I_{ij} = y_i^2 + \sqrt{\sigma^2 + w_{ij}^2}$).Execution Impact: For the Rust engine, an imatrix requires zero code changes. The imatrix does not alter the physical layout of the GGUF file or add new tensors. It is used purely during the pre-processing stage to bias the generation of the S (Scale) and Z (Zero-point) constants, ensuring the lowest reconstruction error for the most critical weights during dequantization.4. Advanced GGUF Branches: I-QuantsFuture-proofing the engine for other builds (like local deployments of TinyTruce agents) might involve parsing I-Quants (e.g., IQ2, IQ3_M).The Architecture: Unlike K-quants which use affine quantization, I-quants use vector quantization. They pack $N$ weights into an $M$-bit code and require lookup tables during decoding.Execution Impact: If loading an IQ model later, expect the inference bottleneck to shift entirely from memory bandwidth to raw CPU compute, as the constant lookup table access heavily taxes the processor cache.Architectural Alignment for the Current BuildThe confirmation that Q4_K_M utilizes dynamic mixed-precision validates exactly why the dual-cache/SWA override was necessary. Forcing the lower SWA layers to ingest the upper Global caches wasn't just a dimensional mismatch; it meant forcing tensors of potentially different internal quantization grades to collide and shatter the attention matrix.
